Practical proactive DDoS-attack mitigation via endpoint-driven in-network traffic control

security

Liu Z, Jin H, Hu Y C, et al. Practical proactive DDoS-attack mitigation via endpoint-driven in-network traffic control[J]. IEEE/ACM Transactions on Networking, 2018, 26(4): 1948-1961.

本文提出主动防御方法，主要是通过一个中间设备mbox执行victim提出的流量策略来保护victim想要的流量，（这篇其实是保护一个云的，一个mbox后面有很多个主机，有的IP是暴露的，有的没有暴露）

Introduction

通过调查许多公司了解到的有：1.他们对学术界提出方法的看法；2.他们当前的防御办法；3.他们对当前使用办法的看法。
本文能解决的问题：1.instant deployability:当前网络的AS太多，而本文的方法布署不需要对当前改变；2.Proactivfe Defense:现有的很多服务（如CSSPs）仍然不好，本文致力于保护victim想要的流量。

  • 过云几十年提出的很多防御办法几乎都无法布署，如：有的方法需要大量软/硬件设备升级。
  • 当前市场DDOS防御提供商：
    1. Cloud Security-Service Providers(CSSPs): 侵犯隐私；只能处理一些明显的攻击。
    2. On-Premise DDoS :Prevention Hardware:DDoS防御功能的硬件。
  • 通过采访安全工程师总结：CSSPs用户有的说可以，大部分仍受攻击，使服务不稳定。由于隐私，一些公司不使用CSSP，（如：medical and goverment institutes,他们与ISP合作，当流量超过指定阈值时过滤掉冒犯流量。）.
  • 调查总结：1.学者提出的方法耗费太大无法布署；2.CSSP只管主要攻击。3.CSSP布署了还是无法阻止有些攻击；4.有的企业由于隐私，无法布署CSSP
  • 设计目标：1.可以在当前网络中布署；2.可以缓解复杂的DDOS攻击；
  • 给目的主机提供细粒的流量控制：当前工作者太片面，无法满足所有的潜在victim需要。本机制可以支持随机受害者流量选择控制，给重要流量特殊通道，经主要客户服务优先级化。目的主机驱动是本机制的关键，victim对自己所需要流量有具体的和完整的认知，因此不管攻击多复杂，只保证交付victim想要的流量即可最小化攻击影响。
  • 攻击模式和假设：攻击者有很多bot，但攻击者不会在mboxs和victim之间。本文机制MiddlePolice(MP)是分布式布署的，，并通过骨干网良好连接，DDOS对mbox攻击不成功。victim与之合作的办法为：victim IP未暴露的则通过capability和其定制的policy进行保护；IP暴露的通过filter进行保护；对多个victim共享path的，使用权重平衡共享。

本机制的结构为：client->mbox->victim;即类似为所有的victim加合作式mbox来保护victim，可以想象成baidu的Server，只有几个IP是暴露的，更多的IP未暴露的主机都是用来进行负载均衡的，当攻击未暴露主机时流量定会通过mbox，mbox再将流量导到victim,这时就有可能多个victim共用一个瓶颈path；当攻击已知IP的主机时就可能绕过mbox，这时使用filter进行保护

mbox的详细设计

流量策略算法：1.探索每个mbox到victim的可用下行带宽；2.分配带宽给发送者并执行victim选择的控制策略。
带宽探索：因为MP不需要布署在下行流链路处，因此对流量可能会高估或低估。解决高估：当包到达mbox时向其内放一个容量戳，当victim的capability handling module(CHM)收到包后便把容量发给mbox，超时没收到就算丢失。根据victim收到包数和丢失率算下行流量；解决低估：mbox给包分类：privileged packets,best-effort packet,使用窗口来发所有privileged包，窗口大小由下行流容量估计和选择策略共同计算。超过的分为best-effort包，mbox交付所有的privileged包，对best-effort包确定一个丢包率，如果无阻塞，best-effort便也全交付。

packet filtering

当前victim的IP保密，攻击者不知其IP则攻击流量必红mbox，当victim 的IP暴露后攻击者可以绕过mbox，这时就要包过滤器来丢弃这些绕过的包。

Traffic filtering primitives

使用基于已存在的ACL配置的机制，mbox将包装入UDP包中，以sPort，dPort为密码验证（32位），mbox与过滤点协商共享密钥。32位被猜对的概率为

Packet Filtering Points

过滤器布置位置：瓶颈link处，如：瓶颈link在innet，则布置在inbound处；在与ISP连接处，则布置在ISP处。

SOURCE AUTHENTICATION

mbox监控TCP连接，并获得hostname。若只是普通主机，同截断并将其重定向到从发送者声明的源地址生成的口令加密主机。发送者收到重定向后便会在host header处用sender-specific hostname，mbox验证该hostname的源IP并将请求交付给victim。该方法可使欺骗的源不产生下行流。若provider可信，victim与provider共享key，否则mbox中继加密连接给victim。
缺陷：只能验证HTTP/HTTPS流，通过civtim验证可能会降低victim的处理能力。

mbox和CHM的实现

mbox和chm基于netFilter Linux Kernel Module实现，软件栈见图2.

过程解释：Client的流量先到mbox进行包装，执行策略，再到CHM进行修正，如果有filter，则还要在th fjtf IP-in-UDP包装，再通过filter的SPORT和DPORT验证。包装过程在[38]中。为防止附加位数导致分片，mbox经过测试设定它的MSS为1360。收到包后CHM获得内容并返回capability（在ACK里面携带，一个ACK可带多个caqpability并且返回信息用IP-IN-IP遂道直接发给mbox，多个capability是放在option里，因此最多可带15个）。从CHM收到ACK后剥去IP头获得capability并更新iTable和cTable，最后重写包的saddr，因为client是与mbox交互的。

形成capability

capability实现约400行，实现在mbox和CHM内核层，使用在AES-NI library上基于CBC-MAC的AES-128算法来计算。mbox形成并验证，CHM只验理论上。

Large scale Evaluation

大尺寸测试，用NS3进行模拟1million到10 million 攻击者洪泛40G链路。所有的攻击者都连到mbox。攻击种类除前两种，还有一种on-off shrew attack[31].攻击者数量比合法用户多10倍。在攻击流不变和shrew攻击中攻击流量为链路容量的3倍，reactive 攻击中每个攻击者开10个连接，合法者开一个。
